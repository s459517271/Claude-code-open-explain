# 04 — 权限安全模型

> Claude Code 不是“模型想调什么就调什么”，每次工具调用前都要先过权限系统

## 为什么这一章很重要

如果没有权限系统，Claude Code 就只是一个把高权限本地能力直接交给模型的危险程序。

真正让它能够落地的关键，不只是工具多，而是：

> 工具在执行前会被严格判定“这次能不能做、该不该问、有没有超范围”。

## 本章关键源码入口

| 文件 | 作用 |
|------|------|
| `src/utils/permissions/PermissionMode.ts` | 定义权限模式 |
| `src/utils/permissions/permissionSetup.ts` | 初始化和切换权限上下文 |
| `src/utils/permissions/permissions.ts` | 规则匹配与权限判定 |
| `src/utils/permissions/pathValidation.ts` | 文件路径安全验证 |
| `src/utils/permissions/denialTracking.ts` | 拒绝记录与追踪 |
| `src/utils/permissions/yoloClassifier.ts` | 自动模式下的分类器逻辑 |

## 新手先记住：权限不是一个简单的 allow / deny

Claude Code 的权限系统更像多层判定流水线：

```text
模型请求调用工具
  ↓
先看当前权限模式
  ↓
再看工具本身的权限语义
  ↓
再看输入参数是否敏感
  ↓
再看路径是否合法
  ↓
再看是否命中允许/拒绝规则
  ↓
最后决定：
  - 自动允许
  - 提示用户确认
  - 直接拒绝
```

所以你不能把它理解成“一个 if 判断”。

## 实际有哪些权限模式

从源码里的 `PermissionMode.ts` 看，至少可以看到这些模式：

- `default`
- `plan`
- `acceptEdits`
- `bypassPermissions`
- `dontAsk`

在某些带条件开关的场景下，还会出现：

- `auto`

### 这些模式可以先这样理解

#### `default`

默认模式。  
读操作通常更容易自动放行，写操作、命令执行、外部动作更容易触发确认。

#### `plan`

计划模式。  
它的核心目标不是高效率，而是先分析、先规划、减少直接改动。

#### `acceptEdits`

从名字就能看出，它比 `default` 更激进，重点是减少编辑类动作上的频繁确认。

#### `bypassPermissions`

几乎就是“绕过常规权限确认”。  
这类模式威力很大，因此也最需要额外谨慎。

#### `dontAsk`

这类模式体现了 Claude Code 内部权限系统不仅服务交互式 CLI，也要服务更自动化、更程序化的调用场景。

#### `auto`

这不是简单的“全部自动允许”，而是和专门的自动模式判定逻辑绑在一起。  
换句话说，自动模式也不是无脑放权。

## 权限模式为什么单独做成系统

因为 Claude Code 需要同时支持多种使用方式：

- 交互式终端
- 更自动化的执行流
- 计划模式
- SDK / 非交互场景
- 多 Agent 协作

如果没有一层统一的权限模式抽象，这些场景会把工具系统撕得很碎。

## 路径校验为什么是权限系统的核心组成部分

新手很容易只盯着“工具名”，但真正危险的往往是“路径”。

例如同样是写文件：

- 写 `src/app.ts`
- 写 `.git/hooks/post-commit`
- 写系统敏感目录

这 3 件事的风险完全不同。

所以 Claude Code 会专门做路径校验，重点关注：

- 是否越过允许目录
- 是否尝试路径逃逸
- 是否命中内部受保护文件
- 是否在自动编辑场景下触碰危险位置

这说明它不是“工具级授权”就结束了，而是一直往参数层深挖。

## 为什么 Bash 命令要走更重的权限链路

因为 Shell 命令是最强、也最危险的工具之一。

同样是 `BashTool`，风险差异极大：

- `pwd`
- `git status`
- `cat README.md`

和：

- `rm -rf`
- `git push --force`
- 重定向写文件
- 管道执行不可信脚本

完全不是一个风险等级。

所以 Claude Code 对命令的处理不会停在“这是 BashTool，所以都一样”，而是继续分析：

- 这是读命令还是写命令
- 是否可能造成破坏
- 是否访问了敏感路径
- 当前模式是否允许这类动作

## 规则系统不是摆设

除了模式本身，Claude Code 还有规则层。

规则层的作用是让权限不只是依赖“当前模式”，还可以根据更细粒度条件判定，例如：

- 某类工具默认允许
- 某类路径总是拒绝
- 某类命令必须询问

这让权限系统从“模式切换”升级成“策略系统”。

## 自动模式为什么还需要分类器

很多人会误以为“自动模式 = 全放开”，源码不是这么设计的。

在自动模式相关逻辑里，你能看到专门的分类器模块，例如 `yoloClassifier.ts`。这说明自动模式并不是放弃判断，而是：

> 把“是否应该自动放行”本身也做成一个需要分析的过程。

这非常关键，因为真正可用的自动化，不是胆子大，而是知道什么时候该停。

## 权限拒绝为什么还要被记录下来

`QueryEngine.ts` 里会持续记录 `permissionDenials`。

这件事的意义至少有 3 层：

1. UI 或 SDK 可以知道到底哪一步被拒绝了
2. 后续系统可以针对拒绝行为做反馈和分析
3. 模型在下一步可能会调整策略，而不是死磕同一个被拒动作

这说明 Claude Code 的权限系统不是“挡住就完了”，而是整个对话状态的一部分。

## 计划模式和普通模式为什么要区分

这其实体现了 Claude Code 对“工作阶段”的理解。

有些时候，用户想要的是：

- 先分析
- 先列计划
- 先解释影响

而不是立刻改代码。

所以 `plan` 这类模式不是性能优化，而是交互语义优化。  
它让系统知道，现在更像“顾问”而不是“直接执行者”。

## 新手应该如何看待这套权限系统

可以把它理解成 3 层防线：

### 第一层：模式

决定大方向，是偏保守、偏计划，还是偏自动。

### 第二层：规则

决定某些工具、路径、命令的细粒度策略。

### 第三层：参数分析

决定这一次具体输入到底是不是危险。

这 3 层叠在一起，Claude Code 才真正具备“既能做事，又不至于失控”的基础。

## 新手常见误区

### 误区 1：权限只是 UI 上那个确认弹窗

不对。弹窗只是最后一层交互表现，前面已经做了很多分析。

### 误区 2：给了某个工具权限，就等于所有同类操作都安全

不对。路径、参数、模式、规则都可能让同一个工具在不同场景下得到不同结论。

### 误区 3：自动模式就是彻底放权

不对。自动模式真正难的地方恰恰是“自动但仍然有判断”。

## 本章小结

Claude Code 的权限系统不是给模型套一个简陋开关，而是：

- 用模式定义大方向
- 用规则补充策略
- 用参数分析处理具体风险
- 用记录机制把拒绝纳入会话状态

这也是它能在真实终端环境里落地的关键前提。
